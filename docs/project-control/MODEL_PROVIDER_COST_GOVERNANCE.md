# Model, Provider, Paid-Service, And Cost Governance

## Provider-Agnostic Rule

Permanent policy is provider-agnostic and evaluation-driven. A UI-visible model label is UI-observed evidence only. Model names, capabilities, pricing, quotas, context limits, effort levels, and availability are current facts only when a current official primary source and retrieval date are recorded. No such vendor-specific fact is newly asserted by this gate.

## Task Classes And Evaluation

Evaluation covers bounded editing, synthesis, deterministic validation, sensitive evidence preparation, multimodal review, and long-context analysis. Candidate routing compares task success, groundedness, rework, escalation rate, latency, privacy handling, availability, and total cost. Allowlists are versioned, time-bounded, and retired when evidence expires or a provider deprecates support.

## Fallback And Portability

Fallbacks must preserve or strengthen data classification and decision gates. There is no uncapped fallback. Provider adapters must keep canonical inputs, outputs, evaluation sets, and audit evidence portable. A fallback that changes risk or data handling requires owner approval.

## Paid-Pilot Reconciliation

Zero spend remains the default before revenue. D21 permits only an explicitly owner-approved, bounded pilot with named objective, provider, data class, maximum monthly budget, duration, measurable exit criteria, and hard shutdown threshold. This harmonizes staged pilots with the zero-spend rule; it does not activate spending.

## Cost Controls

Future paid use requires monthly ceilings, per-run and per-day limits, no automatic recharge, no uncapped retry or fallback, quota-bottleneck evidence, owner approval, and immediate shutdown at the lower of budget or risk threshold. Production cost models must cover daily, weekly, monthly, and annual scenarios, including rework and review cost.
